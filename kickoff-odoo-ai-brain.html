Part 1 · Plain-language overview

This part is written for business owners, managers, and anyone non-technical. No jargon. If you only read one section, read this one.

What we're building, in one sentence

We are setting up Odoo (a complete business management system) and putting an AI assistant on top of it that knows everything inside Odoo and can help anyone in the company through a chat box.

The simple picture

Think of Odoo as a giant filing cabinet

Inside Odoo there are folders for customers, sales, invoices, products, employees, projects, support tickets — everything a business runs on. Today, a person has to open the right drawer, find the right folder, read it, and decide what to do.

Now imagine an assistant who has read every folder

That's the AI. It sits next to the filing cabinet. You ask it a question in plain English — "Which customers haven't paid us in 60 days?" or "Reconcile last week's bank statement" — and it goes into the cabinet, finds the answer, and either tells you or prepares the work for you to approve.

Where you talk to it

There is a chat box inside Odoo, on every screen. You click it, type your question, and the AI answers. If it needs to make a change (like creating a quote or posting an invoice), it shows you the change first and waits for you to click Approve.

What the AI is allowed to do

  • Read any record the user is allowed to see — customers, invoices, stock, tickets, projects, etc.
  • Summarize long lists, complex reports, or message threads.
  • Search using natural language ("show me all overdue invoices from European clients").
  • Draft work — quotes, follow-up emails, journal entries, tasks — but not finalize them without permission.
  • Act, with approval, on important things like posting invoices, sending emails, or reconciling bank lines.

What the AI is NOT allowed to do

  • It cannot see anything the logged-in user is not allowed to see. It uses Odoo's own permission system. There is no backdoor.
  • It cannot post invoices, send emails, or make money-moving changes without a human clicking Approve.
  • It cannot send sensitive data (salaries, bank accounts, personal IDs) to external AI providers when "private mode" is on — it uses a local AI model instead.

Why this is valuable

Most ERPs require training to use. Staff spend hours hunting through menus, exporting to Excel, writing follow-ups, matching payments to invoices. The AI brain collapses that work into a conversation. A junior accountant asks "what's wrong with vendor ABC's balance?" and gets an answer with the supporting evidence in seconds. A salesperson asks "draft a quote like the last one for ACME but with 5% discount" and reviews a finished draft. Time saved is measured in hours per person per day.

How long it takes and what it needs

Build timeRoughly 6 months to a stable production rollout, with the first useful version in about 8 weeks.
Team1 functional consultant, 2 Odoo developers, 1 AI engineer, 1 DevOps, 1 product owner.
ServersOne server for Odoo, one for the AI service, optionally one with a graphics card if we want a private/local AI.
External servicesAn OpenAI or Anthropic account (paid per usage). Optional: a self-hosted model for private data.

What can go wrong (and how we handle it)

ConcernHow we address it
"The AI makes a mistake and ruins our data."Every AI-made change is tagged. High-impact changes need human approval. We keep a log of everything and can roll back.
"It exposes our customer data to OpenAI."We can run in private mode using a local AI. Sensitive fields are hidden before any data leaves our servers.
"It costs too much in API fees."We set monthly budgets per user, route simple questions to a cheap model, and cache common answers.
"It tells users wrong information confidently."Every answer shows the source records it used, so users can verify in one click.

Existing projects we are building on top of

We are not starting from zero. There are already two solid foundations we will use:

  1. Odoo's own AI app (built into Odoo 19) — gives us the chat panel and a basic agent system.
  2. Apexive's open-source odoo-llm framework — gives us provider switching (OpenAI, Anthropic, local), a tool system, and pre-made accounting tools.

Our custom work focuses on the parts those don't cover: stronger approval and audit, a private-mode redaction layer, a cleaner chat UX, and any industry-specific tools the client needs.

Part 2 · How to use this document with Claude Code

Claude Code is a tool that lets you delegate coding tasks from your terminal or desktop. To kick off this project, you give it the brief from Part 3 of this document. It will then create the project structure, install dependencies, and begin building according to the sequence.

Step-by-step

1

Prepare an empty folder

On your computer, create a folder for the project, for example ~/projects/odoo-ai-brain. Open a terminal in that folder.

2

Start Claude Code in that folder

Run claude in the terminal (or open the Claude Code desktop app and point it at the folder). You should see a prompt waiting for input.

3

Copy the brief from Part 3

Scroll to Part 3 below. Copy the entire text inside the dashed box. Paste it into Claude Code as your first message and send.

4

Let Claude Code plan, then approve

Claude Code will read the brief, propose a project structure and a plan, and ask you to confirm before it starts creating files. Review the plan, ask follow-up questions if anything is unclear, then approve.

5

Provide secrets when asked

Claude Code will ask for an OpenAI or Anthropic API key, and optionally an Odoo Enterprise subscription key. Have these ready. Never paste them into the chat — store them in a local .env file as Claude Code will instruct.

6

Work milestone by milestone

The brief tells Claude Code to work in numbered milestones. After each milestone it should stop, summarize what it did, and wait for you to test before continuing. Do not skip the testing — each milestone builds on the last.

Important: Treat Claude Code as a senior developer pairing with you, not a magic box. Read its plans, ask questions, run the tests it writes, and push back when something looks off. The brief in Part 3 explicitly tells Claude Code to work this way.

===== BEGIN BRIEF =====

ROLE
You are the lead engineer for a project called "Odoo AI Brain." You will
plan, scaffold, and implement the project in this empty repository, working
milestone by milestone. After each milestone, stop and wait for human
review before continuing.

PROJECT GOAL
Set up Odoo 19 and add an AI layer on top of it. The AI must be able to
read all apps and records the logged-in user is permitted to see, draft
and (with approval) execute actions, and converse with the user through a
chat panel embedded in the Odoo web client. Every action must be logged,
reversible where possible, and obey Odoo's existing permission system.

NON-NEGOTIABLES
1. The AI never has more privilege than the user it is acting for. It
   authenticates as that user via short-lived JWTs and goes through Odoo's
   JSON-RPC API. No super-user backdoor.
2. Write actions of medium or high impact require explicit user approval
   in the chat UI before execution.
3. Every AI-initiated read/write is recorded in an audit log
   (model: ai.tool.log) with inputs, outputs, latency, and approver.
4. Sensitive fields (configurable allowlist) must be redacted before any
   payload is sent to an external LLM provider, unless the user is in the
   group permitted to see them.
5. The system supports a "private mode" tenant flag that forces routing
   to a local Ollama model and disables all external LLM calls.
6. We build ON TOP of two existing projects rather than reinventing them:
   - Odoo 19's native AI app (productivity.ai) for the chat surface and
     base agent framework.
   - Apexive's open-source odoo-llm framework
     (https://github.com/apexive/odoo-llm) for provider abstraction, tool
     framework (@llm_tool decorator), MCP server, and the accounting tool
     pack (llm_tool_account).
   Our custom module ai_brain extends, glues, and hardens these layers.

TARGET STACK
- Odoo 19 (Community is fine for development; Enterprise if license is
  available).
- PostgreSQL 16 with the pgvector extension.
- Python 3.11 for both Odoo and the orchestrator.
- FastAPI for the orchestrator service.
- Redis for sessions, rate limits, background jobs.
- LLM providers: OpenAI (default cloud), Anthropic (alternate), Ollama
  (local). Use LiteLLM-style routing.
- Embeddings: text-embedding-3-small for cloud, bge-m3 for local.
- Frontend: Odoo's OWL 2 framework for the chat widget.
- Containers: Docker Compose for development. Provide a Helm chart skeleton
  for production but do not flesh it out yet.
- Observability: structured JSON logs, OpenTelemetry traces, Prometheus
  metrics endpoints.

4.3 Tool registry contract

Every tool exposed to the AI must declare:

FieldPurpose
nameUnique stable identifier, e.g. account.reconcile_proposal.
descriptionPlain English; the LLM uses this to decide when to call it.
input_schemaJSON Schema for arguments. Strict validation; reject extra fields.
output_schemaJSON Schema for the result.
required_groupsOdoo groups the calling user must belong to.
requires_approvalIf true, execution pauses and surfaces an Approve/Reject in the chat.
side_effectsnone | read | write | external.
rate_limitCalls per minute per user.

4.4 Request flow (canonical)

  1. User types in the chat panel; Odoo controller /ai_brain/chat persists an ai.message and issues a JWT.
  2. Orchestrator receives prompt + JWT + context (res_model, res_id).
  3. Orchestrator loads conversation history and runs RAG to fetch relevant chunks.
  4. LLM is called in a planning loop (ReAct): Thought → Action (tool call) → Observation → repeat, capped at 8 steps.
  5. Each tool call goes through Odoo JSON-RPC as the user; redacted fields are stripped before payloads leave the orchestrator.
  6. Write tools tagged requires_approval stream a pending approval card to the UI; user clicks Approve to resume.
  7. Final natural-language answer streams back to the chat. ai.tool.log entries are written for every step.

Part 6 · Acceptance checklist

Before we declare the project ready for production, every item on this list must be true.

Functional

  • Chat panel available on every Odoo screen for authenticated users.
  • AI can answer questions across at least Sales, Accounting, Inventory, Helpdesk.
  • Approval flow works for all write tools.
  • Conversation history persists per user and is searchable.
  • RAG returns sources (record links) with every answer that uses them.
  • UI language matches the user's Odoo language setting.

Security

  • Penetration test passed (prompt injection, IDOR, JWT replay, rate limit bypass).
  • Redaction unit tests cover every sensitive field configured.
  • Private mode verifiably blocks all external LLM calls.
  • Every AI write is tagged with x_ai_origin and a revert tool exists.

Operational

  • Grafana dashboard shows tokens, costs, latency, errors per provider and per user.
  • Runbook covers: provider outage, rate-limit storm, bad prompt template, vector store rebuild.
  • Backup and restore tested for both Odoo DB and vector data.
  • make eval passes ≥ 95% of the golden task set.

Documentation

  • README lets a fresh developer reach M1 acceptance unaided.
  • User playbooks exist per role with screenshots.
  • Tool registry document is current and matches code.
  • At least 3 short training videos recorded for end users.
